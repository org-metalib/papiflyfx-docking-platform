Known defects from the review
1. The generated root `pom.xml` is invalid because the root template uses `${rootArtifactId}-app` in `<modules>`, while the archetype descriptor generates the child module into `app/`.

4. The archetype has no integration-test project under `src/test/resources/projects`, so `mvn install` warns that no archetype IT projects exist.

Task
Fix the archetype so it generates a buildable starter project and document the correct usage.

Required outcomes
- A freshly generated project contains exactly one root module path: `app`.
- The child app module remains clearly identified in docs and commands.
- Generated Markdown docs preserve their headings and structure.
- `papiflyfx-docking-archetype` has at least one archetype integration test project.
- Validation results are recorded in `spec/papiflyfx-docking-archetype/progress.md` or `validation.md`.

- Prefer keeping the physical module directory as `app/`.
- It is acceptable for the child artifactId to remain `${rootArtifactId}-app`, but if you keep that pattern, explain it explicitly in docs.
- Update all docs to use one consistent convention:
  - directory path: `app/`
  - command selector: either `-pl app` or `-pl :${rootArtifactId}-app`
- For Markdown templates, avoid raw `##` / `###` in filtered Velocity files unless you escape `#` safely.
- Do not change unrelated framework modules unless validation proves they are required for the fix.
