# PapiflyFX App Init Prompt

Use this prompt in Codex, Claude, or Copilot to start the archetype/bootstrap task from the repository's existing agent and spec workflow.

```text
As @ops-engineer with @spec-steward review, start the PapiflyFX application bootstrap initiative.

Read these first:
- AGENTS.md
- README.md
- pom.xml
- spec/agents/README.md
- spec/agents/playbook.md
- spec/agents/ops-engineer.md
- spec/agents/spec-steward.md
- spec/papiflyfx-docking-archetype/README.md

Follow the repository's single-lead model, review gates, spec-first workflow, and handoff contract.

Task
Create a detailed plan for a new Maven archetype module that bootstraps a multi-module PapiflyFX application project which mirrors this framework's Maven setup and developer workflow.

The generated application starter should include:
- a parent multi-module Maven build
- a runnable `main` JavaFX application module
- Java and JavaFX versions aligned with the PapiflyFX framework versions selected in this repository
- Maven wrapper support
- GitHub Actions for CI/CD
- a spec-first, multi-agent setup for Codex, Claude, and Copilot
- clear environment setup and application startup instructions

Also evaluate and recommend adding a `papiflyfx-docking-bom` module so generated applications can import a PapiflyFX BOM instead of repeating version declarations.

Repository baseline to mirror
- Java: 25
- SDKMAN example: `sdk use java 25.0.1.fx-zulu`
- JavaFX: 25.0.2
- Maven wrapper baseline: Maven 3.9.12 via `./mvnw`
- Existing build conventions: root-managed `dependencyManagement`, `pluginManagement`, `maven-enforcer-plugin`, `javafx-maven-plugin`, `os-maven-plugin`, headless TestFX defaults

Scope expectations
- plan the new archetype module itself
- define the generated project/module structure
- define the generated multi-agent documentation layout
- define the generated GitHub workflow files
- define how version alignment works between the framework, BOM, and generated app
- define how a developer sets up the environment, generates the project, builds it, tests it, and starts the main application

Generated project expectations
- root aggregator `pom.xml`
- `main` application module with a minimal PapiflyFX JavaFX entry point
- shared dependency management through either a new PapiflyFX BOM or a clearly justified alternative
- Maven wrapper scripts and wrapper properties
- `.github/workflows/ci.yml` and any release/publish workflow skeleton you consider necessary
- `AGENTS.md`, `CLAUDE.md`, `.github/copilot-instructions.md`, and a small `spec/` skeleton for research/plan/progress/validation
- starter README with setup, build, test, and run instructions

Work location
- Use `spec/papiflyfx-docking-archetype/` for task artifacts created during planning.
- Create or update `research.md`, `plan.md`, `progress.md`, and `validation.md` there if the task needs them.

Constraints and guidance
- Do not invent versions independently; use this repository's root POM as the source of truth and call out exact values you find.
- Prefer the same Maven patterns already used here instead of introducing a different build philosophy.
- Keep the plan compatible with JavaFX platform handling already used by this repository.
- Bias toward a starter project that is immediately buildable with `./mvnw`.
- The generated app should start with a single command such as `./mvnw -pl main javafx:run`, unless you recommend a better module name and explain why.
- Include headless test guidance for CI and local verification.
- Keep the generated project small, opinionated, and easy to extend.

Return
1. Intake summary with priority, lead, reviewers, impacted modules, and key invariants
2. Research summary of the current PapiflyFX build conventions that the archetype must preserve
3. Detailed implementation plan for the archetype module and any supporting BOM module
4. Proposed generated project layout with module names and file skeleton
5. Environment setup instructions for a developer on a fresh machine
6. Exact build, test, and startup commands for the generated application
7. GitHub Actions CI/CD proposal with workflow names, triggers, and main steps
8. BOM recommendation with rationale, dependency-management shape, and sample import usage
9. Validation strategy for both the archetype and the generated starter app
10. Risks, assumptions, and follow-up tasks

Where useful, include short POM snippets, workflow YAML snippets, and example generated commands.

Do not jump straight into full implementation. Start with the detailed plan and environment/startup instructions first, grounded in the actual repository structure and current versions.
```
